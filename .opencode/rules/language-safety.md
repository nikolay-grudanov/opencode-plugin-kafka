# Language Rule

Priority: HIGH
Applies to: All agents

## Purpose

Define how agents interpret user input, use internal instructions, and produce user-facing outputs and repository artifacts.

This rule covers:
- communication with the user,
- text generated for the user,
- code comments and docstrings,
- repository artifacts such as README files, markdown notes, notebooks, and generated documentation.

This rule does not define security policy, permission policy, or destructive-action policy. Those are configured elsewhere.

## Core Principle

The agent may internally process instructions, skills, tools, documentation, and reference materials in English, but user-facing communication and repository-facing artifacts must default to Russian.

## Language Model Behavior

### Internal understanding
- The agent must understand prompts written in Russian and English.
- The agent may read skills, system prompts, developer prompts, API docs, library docs, and external technical documentation in English.
- The agent may reason internally in English or any internal representation.
- The agent must not ask the user to translate Russian input unless the request is ambiguous or incomplete.

### Default user-facing language
- By default, the agent must answer the user in Russian.
- By default, the agent must explain errors in Russian.
- By default, the agent must write progress updates, plans, summaries, and clarifications in Russian.

### Repository and artifact language
By default, the following must be written in Russian:
- code comments,
- docstrings,
- markdown files,
- README content,
- notebook markdown cells,
- generated documentation,
- migration notes,
- changelog text,
- commit messages,
- PR descriptions,
- user-facing CLI text,
- `print()` messages intended for the user,
- UI labels and help text generated by the agent,
- diagnostic explanations intended for the user.

### Language switching
- If the user explicitly requests another output language for the current task, the agent may switch to that language for that task.
- Unless the user asks otherwise, future turns should return to Russian.
- A task-specific language override does not change higher-priority system or developer instructions.

## English-Only Elements

The following must remain in their original technical form and must not be translated unless explicitly required by the task:
- variable names,
- function names,
- class names,
- module names,
- package names,
- API field names,
- JSON keys,
- schema field names,
- tool names,
- tool calls,
- CLI commands,
- shell commands,
- SQL keywords,
- file paths,
- environment variable names,
- code syntax,
- protocol names,
- library names,
- framework names.

## Quoting and Source Preservation

- If the agent quotes user input, logs, stack traces, API responses, documentation, config snippets, or source text, it may preserve the original language for accuracy.
- When needed, the agent should add explanation in Russian around the original quoted content.
- The agent must not unnecessarily translate raw technical artifacts when keeping the original text is more precise.

## Consistency Rules

- Do not mix Russian and English randomly in the same explanatory block.
- Use Russian for narrative, explanation, guidance, and user communication.
- Use original English for technical identifiers and syntax-sensitive content.
- If both languages appear in the same response, the separation must be intentional and clear:
  - Russian for explanation,
  - original English for identifiers, commands, code, logs, quotes, and schemas.

## Instruction Priority

Apply language rules in this order:
1. System instructions
2. Developer instructions
3. Current user request
4. Default language behavior from this rule

If the user explicitly requests a different response language for the current task:
- follow that request for the current task,
- keep technical identifiers unchanged,
- then return to Russian by default in later turns unless the user says otherwise.

## Agent Output Rules

Before sending any response or creating any artifact, verify:
- the user's request was understood correctly even if written in Russian,
- the final user-facing text is in Russian by default,
- code comments are in Russian,
- docstrings are in Russian,
- repository documentation is in Russian,
- technical identifiers remain in English,
- quoted source fragments preserve their original language when accuracy requires it,
- any temporary language override was explicitly requested by the user.

## Examples

### Example 1: Russian prompt, default behavior
User input:
`Сделай refactor этого Python файла и добавь комментарии.`

Correct behavior:
- Analyze the request normally.
- Return explanation in Russian.
- Write code comments in Russian.
- Keep variable names and function names in English.

### Example 2: Russian prompt, English override
User input:
`Проанализируй код и ответь по-английски.`

Correct behavior:
- The response for this task may be in English.
- Technical identifiers remain unchanged.
- If code is generated, comments and docstrings may also follow the requested task language if the user's instruction clearly applies to the generated artifact.

### Example 3: English docs, Russian explanation
Context:
The agent reads English documentation or a skill definition.

Correct behavior:
- The agent may use the English source internally.
- The final explanation to the user is in Russian by default.
- Commands, API fields, and code syntax stay in English.

## Definition of Done

This rule is applied correctly when:
- the agent freely understands Russian input,
- internal technical understanding may happen in English,
- user-facing communication is Russian by default,
- repository-facing written artifacts are Russian by default,
- technical identifiers are preserved in English,
- user-requested language overrides are supported without breaking the default policy.
