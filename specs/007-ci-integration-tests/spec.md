# Feature Specification: CI Integration Tests Pipeline

**Feature Branch**: `007-ci-integration-tests`  
**Created**: 2026-04-24  
**Status**: Draft  
**Input**: User description: "Расширить CI pipeline чтобы integration-тесты с контейнеризованным брокером сообщений запускались в CI на каждый PR и push в main."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Разработчик получает обратную связь по integration-тестам при каждом PR (Priority: P1)

Как разработчик, создающий pull request в main, я хочу чтобы integration-тесты запускались автоматически в CI, чтобы сразу видеть, не сломал ли мой код взаимодействие с брокером сообщений.

**Why this priority**: Это основная ценность фичи — без автоматического запуска integration-тестов в CI, ошибки взаимодействия с брокером обнаруживаются только локально или в production. Автоматический запуск на каждый PR — это минимально жизнеспособный продукт.

**Independent Test**: Можно полностью протестировать, создав PR и убедившись, что job с integration-тестами запустился, прошёл и отобразил результат в PR checks.

**Acceptance Scenarios**:

1. **Given** открыт PR в ветку main, **When** CI pipeline запускается, **Then** запускается отдельный job с integration-тестами параллельно с остальными проверками
2. **Given** integration-тесты в PR проходят успешно, **When** разработчик смотрит на PR checks, **Then** видит зелёный статус для integration-тестов
3. **Given** integration-тесты в PR падают, **When** разработчик смотрит на PR checks, **Then** видит красный статус с деталями ошибки

---

### User Story 2 — Integration-тесты запускаются параллельно с остальными проверками (Priority: P2)

Как член команды, я хочу чтобы integration-тесты выполнялись параллельно с unit-тестами, lint и build, чтобы общее время CI не увеличивалось линейно.

**Why this priority**: Параллельное выполнение обеспечивает быструю обратную связь. Однако даже последовательное выполнение уже ценно — параллельность это оптимизация.

**Independent Test**: Можно протестировать, запустив CI pipeline и убедившись, что оба job стартовали одновременно и не ждут друг друга.

**Acceptance Scenarios**:

1. **Given** push в main или PR в main, **When** CI pipeline стартует, **Then** job с unit-тестами и job с integration-тестами запускаются одновременно, без зависимости одного от другого
2. **Given** job с unit-тестами падает, **When** integration job ещё выполняется, **Then** integration job продолжает работать и не отменяется

---

### User Story 3 — Integration-тесты защищены от зависания (Priority: P3)

Как разработчик, я хочу чтобы integration-тесты имели ограничение по времени выполнения, чтобы зависший контейнер или тест не блокировал CI pipeline бесконечно.

**Why this priority**: Защита от зависаний важна для стабильности CI, но это не первостепенная ценность — без неё тесты всё равно будут работать корректно в большинстве случаев.

**Independent Test**: Можно протестировать, проверив конфигурацию CI pipeline на наличие таймаута для integration-тестов.

**Acceptance Scenarios**:

1. **Given** integration-тесты запущены, **When** время выполнения превышает установленный лимит, **Then** job автоматически отменяется с соответствующим статусом
2. **Given** нормальное выполнение integration-тестов, **When** все тесты проходят за допустимое время, **Then** job завершается корректно без вмешательства таймаута

---

### Edge Cases

- Что произойдёт если Docker daemon недоступен на runner? Тесты должны быть пропущены (graceful degradation), а не падать.
- Что произойдёт если контейнер брокера сообщений не стартует за разумное время? Таймаут job должен прервать выполнение.
- Что произойдёт при одновременном запуске нескольких PR? Конкурентность (concurrency group) уже настроена в pipeline — каждый PR получает свой запуск.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: CI pipeline ДОЛЖЕН содержать отдельный job для запуска integration-тестов, независимый от job с unit-тестами, lint и build
- **FR-002**: Оба job ДОЛЖНЫ запускаться параллельно при каждом push в main и при каждом pull request в main
- **FR-003**: Job с integration-тестами ДОЛЖЕН иметь таймаут в 15 минут на уровне всего job
- **FR-004**: Job с integration-тестами ДОЛЖЕН автоматически поднимать контейнер брокера сообщений через механизм testcontainers — без ручной настройки Docker Compose
- **FR-005**: CI pipeline ДОЛЖЕН запускать integration-тесты на стандартном облачном runner (ubuntu-latest) без требования self-hosted runner
- **FR-006**: Версии сторонних actions в обоих job ДОЛЖНЫ совпадать (единый pinned SHA)
- **FR-007**: Устаревший комментарий-заглушка о невозможности запуска integration-тестов в CI ДОЛЖЕН быть удалён из существующего job и заменён на актуальное описание
- **FR-008**: Скрипт для запуска integration-тестов ДОЛЖЕН быть определён в конфигурации проекта (package.json) и доступен через стандартную команду запуска

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: При каждом PR и push в main запускаются ровно 2 параллельных job — один с unit-тестами/lint/build, второй с integration-тестами
- **SC-002**: Integration-тесты завершают��я (успехом или неудачей) в пределах 15 минут в 100% случаев
- **SC-003**: Все существующие integration-тесты (6 сценариев) проходят в CI на стандартном облачном runner без дополнительных настроек
- **SC-004**: Время обратной связи разработчика не увеличивается — integration-тесты выполняются параллельно с unit-тестами
- **SC-005**: При успешном выполнении всех тестов оба job показывают зелёный статус в PR checks

## Assumptions

- Стандартный облачный runner (ubuntu-latest) имеет предустановленный Docker daemon, достаточный для запуска testcontainers
- Существующая конфигурация integration-тестов (vitest config, testcontainers setup) корректна и не требует изменений
- Существующие integration-тесты (6 сценариев) уже написаны и проходят локально
- Механизм graceful degradation в тестах корректно обрабатывает недоступность Docker — тесты пропускаются, а не падают
- Конкурентность (concurrency group) в CI pipeline настроена корректно и не требует изменений
- Скрипт test:integration уже присутствует в package.json (подтверждено анализом кодовой базы)
