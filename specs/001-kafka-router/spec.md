# Feature Specification: Kafka Router Plugin — Configuration & Message Routing

**Feature Branch**: `001-new-specification`
**Created**: Tue Apr 21 2026
**Status**: Draft
**Input**: User description: "Создание Kafka router плагина с Zod-валидацией конфигурации и функциями маршрутизации сообщений по правилам"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Администратор настраивает правила маршрутизации (Priority: P1)

Администратор плагина создаёт JSON-конфигурацию с набором правил, определяющих:
- К какому Kafka-топику привязано каждое правило
- Какое условие (фильтр) должно быть применено к сообщению
- Какой агент должен обработать сообщение
- Какую команду использовать при обработке

**Why this priority**: Без корректной конфигурации плагин не сможет функционировать. Это базовый сценарий, без которого вся система бессмысленна.

**Independent Test**: Можно протестировать отдельно, создав валидный JSON-конфиг и проверив, что он корректно парсится в объекты.

**Acceptance Scenarios**:

1. **Given** валидный JSON с `topics` и `rules`, **When** вызывается `parseConfig(rawJson)`, **Then** возвращается строго типизированный объект `PluginConfig` без ошибок
2. **Given** JSON с отсутствующим обязательным полем `agent` в правиле, **When** вызывается `parseConfig(rawJson)`, **Then** выбрасывается `ZodError` с описанием ошибки
3. **Given** JSON с отсутствующим полем `prompt_field`, **When** вызывается `parseConfig(rawJson)`, **Then** используется значение по умолчанию `"$"` для всех правил

---

### User Story 2 — Система маршрутизирует сообщение агенту (Priority: P1)

При получении сообщения из Kafka-топика система должна определить:
- Какое правило applies к данному топику
- Удовлетворяет ли сообщение условию правила
- Какой агент должен обработать сообщение
- Какой промпт сформировать для агента

**Why this priority**: Это ключевая функциональность плагина — маршрутизация сообщений правильным агентам.

**Independent Test**: Можно протестировать отдельно, подав тестовый payload и набор правил, проверив возвращённый промпт.

**Acceptance Scenarios**:

1. **Given** сообщение с `vulnerabilities` массивом, содержащим элемент с `severity: "CRITICAL"`, **When** вызывается `matchRule(payload, "security", rules)`, **Then** возвращается правило с `condition: "$.vulnerabilities[?(@.severity=='CRITICAL')]"`
2. **Given** сообщение без критических уязвимостей, **When** проверяется то же правило, **Then** возвращается `null` (правило не применяется)
3. **Given** правило без `condition`, **When** вызывается `matchRule` с любым сообщением этого топика, **Then** возвращается это правило (catch-all поведение)

---

### User Story 3 — Формирование промпта для агента (Priority: P1)

Система извлекает данные из сообщения и формирует текст запроса для агента.

**Why this priority**: Без корректного промпта агент не получит нужную информацию для обработки.

**Independent Test**: Тестируется независимо от маршрутизации — можно подать payload и правило, проверить итоговую строку.

**Acceptance Scenarios**:

1. **Given** сообщение с полем `task_text: "Audit the code"`, **When** вызывается `buildPrompt` с `prompt_field: "$.task_text"` и `command: "audit"`, **Then** возвращается строка `"/audit Audit the code"`
2. **Given** сообщение с вложенным JSON-объектом в поле `details`, **When** вызывается `buildPrompt` с `prompt_field: "$.details"`, **Then** возвращается сериализованный JSON в виде строки
3. **Given** пустой payload и правило без `prompt_field`, **When** вызывается `buildPrompt`, **Then** используется fallback-строка `"Process this payload"`

---

### Edge Cases

- Что происходит, когда `payload` равен `null` или `undefined`?
- Что происходит, когда `jp.query` возвращает пустой массив?
- Что происходит, когда извлечённое значение — число, а не строка?
- Что происходит, когда в конфиге несколько правил для одного топика?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Плагин должен предоставлять функцию `parseConfig(rawJson: unknown)` для валидации JSON-конфигурации
- **FR-002**: Функция `parseConfig` должна возвращать объект типа `PluginConfig` или выбрасывать `ZodError` при некорректной конфигурации
- **FR-003**: Схема `RuleSchema` должна содержать обязательные поля: `name`, `topic`, `agent` и опциональные: `condition`, `command`, `prompt_field`
- **FR-004**: Схема `PluginConfigSchema` должна содержать обязательные поля: `topics` (минимум 1 элемент) и `rules` (минимум 1 элемент)
- **FR-005**: Поле `prompt_field` должно иметь значение по умолчанию `"$"` при отсутствии в конфигурации
- **FR-006**: Функция `matchRule(payload: unknown, topic: string, rules: Rule[])` должна фильтровать правила по топику и применять JSONPath-условия
- **FR-007**: Функция `matchRule` должна возвращать первое совпавшее правило или `null`, если ни одно не подошло
- **FR-008**: Функция `buildPrompt(payload: unknown, rule: Rule)` должна извлекать текст из payload по `prompt_field` и формировать итоговую строку
- **FR-009**: Если в правиле задан `command`, итоговая строка должна начинаться с `/{command}`
- **FR-010**: Если извлечённое значение — объект или массив, оно должно быть сериализовано через `JSON.stringify()`
- **FR-011**: При отсутствии данных для извлечения должна использоваться fallback-строка `"Process this payload"`

### Key Entities

- **PluginConfig**: Корневой объект конфигурации плагина, содержащий `topics` и `rules`
- **Rule**: Правило маршрутизации с полями `name`, `topic`, `condition`, `agent`, `command`, `prompt_field`
- **Payload**: Произвольный JSON-объект сообщения из Kafka-топика

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Валидный JSON-конфиг парсится без ошибок и возвращает типизированный объект
- **SC-002**: Невалидный JSON вызывает `ZodError` с понятным сообщением, включающим путь к полю, ожидаемый тип и фактическое значение
- **SC-003**: Правило с JSONPath-условием корректно фильтрует сообщения по критерию
- **SC-004**: Правило без условия применяется ко всем сообщениям своего топика
- **SC-005**: Промпт корректно формируется со slash-командой и извлечённым текстом
- **SC-006**: Вложенные JSON-объекты корректно сериализуются в строку
- **SC-007**: Unit-тесты покрывают все краевые случаи: успешный парсинг, ошибки валидации, условия матчинга, формирование промпта

## Assumptions

- Предполагается, что JSONPath-библиотека (jsonpath) доступна и корректно обрабатывает выражения
- Предполагается, что используется Zod для валидации схем
- Предполагается, что тесты запускаются через Vitest
- Предполагается, что структура файлов создаётся внутри `src/` директории плагина
